## 🔴 CRITICAL — Harus Diperbaiki Segera

### #1 — XSS Vulnerability di Kurasah Editor

**Lokasi:** `src/markdown-render.jsx` line 6-111 + `src/pages/kurasah-editor.jsx` line 199 (`dangerouslySetInnerHTML`)

**Masalah:** Markdown renderer **tidak meng-escape HTML user input**. Kalau user nulis di catatan:
```
<script>alert('XSS')</script>
<img src=x onerror="fetch('/api/steal?cookie='+document.cookie)">
```
…itu akan **execute** saat preview di-render. Karena ini single-user localStorage app, dampak terbatas, **tapi:**
- User bisa accidentally nulis HTML dari paste (misal dari kitab online yang punya HTML)
- Kalau nanti ada feature import/export atau share antar user, ini langsung jadi attack vector
- Ini melanggar prinsip "least surprise" — markdown harusnya aman

**Fix:**
```js
const escapeHtml = (s) => s.replace(/[&<>"']/g, (c) => ({
  "&":"&amp;","<":"&lt;",">":"&gt;",'"':"&quot;","'":"&#39;"
}[c]));

// Lalu di setiap line processing, escape DULU sebelum apply markdown:
const inline = (text) => {
  return escapeHtml(text)
    .replace(/\*\*(.+?)\*\*/g, '<strong class="...">$1</strong>')
    // ...dst
};

// Di blockquote arabic, paragraf, semua tempat yang inject teks user → escape dulu.
```

**Prioritas: 🔴 sangat tinggi.** Ini fix 15 menit, dampaknya besar.

### #2 — Admin PIN Hardcoded & Ditampilkan di UI

**Lokasi:**
- `src/data.jsx:614` → `const ADMIN_PIN = "9090";`
- `src/pages/admin.jsx:91` → menampilkan PIN sebagai "Demo PIN: 9090" di login screen

**Masalah:**
- Siapapun yang akses `/admin` (link-nya bahkan **ada di footer publik** — `src/layout.jsx:170`) bisa lihat PIN dan masuk
- Admin panel punya akses ke: generate member code, lihat semua data onboarding, edit guide, settings
- Ini bukan demo project — repo udah ada Lynk.id integration untuk payment

**Fix:**
1. Hapus baris "Demo PIN:" dari admin login screen
2. Hapus link "Admin" dari footer publik (akses via direct URL saja)
3. Ganti PIN dari "9090" ke yang lebih panjang dan random
4. Idealnya: simpan PIN di-hash di constant (SHA-256), compare dengan hash di runtime. Bukan plaintext.
5. Long-term: tambahkan rate limiting (max 5 attempt per 10 menit) dan log audit attempts

**Prioritas: 🔴 sangat tinggi.** Production blocker.

### #3 — Payment Modal Auto-Confirm Tanpa Verifikasi

**Lokasi:** `src/layout.jsx:298-301`

**Masalah:**
```js
const handlePayClick = () => {
  window.open("https://lynk.id/madad", "_blank", "noopener,noreferrer");
  setTimeout(() => setPaid(true), 800);  // ← 800ms setelah klik, auto "paid" state
};
```

User klik tombol "Bayar via Lynk.id" → tab Lynk terbuka. 800ms kemudian modal otomatis berubah ke screen "Terima kasih! Pembayaran sedang diproses" — **terlepas dari apakah user benar-benar bayar**.

Implikasi:
- User bisa close tab Lynk tanpa bayar, modal terus nunjukin "Pembayaran sedang diproses"
- Ini misleading — user mungkin nunggu kode admin, padahal admin gak akan dapat notifikasi karena gak ada transaksi
- User experience kacau: "Loh kok belum dikirim kode-nya ya?"

**Fix:** Ganti `setTimeout(() => setPaid(true), 800)` jadi state baru "menunggu konfirmasi", dan munculkan tombol manual "Saya sudah bayar, lanjut" yang user klik sendiri **setelah benar-benar selesai checkout di Lynk**. Atau lebih baik: pakai Lynk webhook untuk konfirm otomatis (kalau Lynk support).

**Prioritas: 🔴 tinggi.** Mengganggu trust flow.

## 🟡 IMPORTANT — Anti-Pattern & Gap dari Design.md

### #4 — Emoji 🎉 di Onboarding & Paths (Anti-Pattern Sanctuary)

**Lokasi:**
- `src/pages/onboarding.jsx:77` → `toast.push("Dashboard personal sudah siap 🎉");`
- `src/pages/paths.jsx:20` → `toast.push(\`"${module.title}" selesai 🎉\`);`
- `src/pages/paths.jsx:128` → `<div className="text-5xl mb-3">🎉</div>` (saat path 100%)

**Masalah:** Emoji 🎉 (party popper) adalah **classic SaaS celebration pattern**. Ini melanggar mood `calm, intelligent, focused, subtly spiritual` di design.md. Sanctuary tidak meledak-meledak ketika user menyelesaikan sesuatu.

**Fix:** 
- Onboarding finish → `toast.push("Dashboard personal sudah siap untukmu.")`
- Module selesai → `toast.push("'${title}' tercatat selesai.")` atau pakai Arabic appropriate `بَارَكَ اللهُ فِيكَ` di toast (subtle gold)
- Path 100% completion screen: ganti 🎉 dengan **ornamen kaligrafi** (Arabic completion phrase: `الْحَمْدُ لِلَّهِ الَّذِي بِنِعْمَتِهِ تَتِمُّ الصَّالِحَاتُ` — "Segala puji bagi Allah yang dengan nikmat-Nya segala kebaikan menjadi sempurna"). Atau ornamen geometric medallion.

**Prioritas: 🟡 tinggi.** Ini melanggar prinsip explicit di prompt sanctuary.

### #9 — Script Loading Order Fragile

**Lokasi:** `index.html` lines 219-241

**Masalah:** `dashboard.jsx` (line 226) di-load **sebelum** `dashboard-companions.jsx` (line 240). Karena `dashboard.jsx` reference `IntentionStrip`, `RhythmCard`, `ReflectionCard` di JSX, JavaScript hoisting + window-attached pattern bikin ini **kebetulan jalan** (karena reference di evaluate saat React render, bukan saat script execute). Tapi ini fragile — di browser tertentu atau dengan timing tertentu bisa error.

Sama untuk `paths.jsx` (line 228) yang reference `markPresenceToday`.

**Fix:** Reorder script tags supaya dependency lower-level di-load duluan:
```
1. icons, ui, data, auth, layout (foundation)
2. muqaranah-data, intentions-data (data layers)
3. markdown-render, quick-note, dashboard-companions (shared components)
4. pages/* (yang reference semua di atas)
5. app.jsx (last)
```

**Prioritas: 🟡 sedang.** Fix sebelum bug random terjadi di production.

### #14 — `useMemoC` di ReflectionCard Tidak Re-trigger saat Notes Berubah

**Lokasi:** `src/dashboard-companions.jsx:410-412`

```js
const note      = useMemoC(() => getReflectionOfTheDay(), []);
const allNotes  = useMemoC(() => loadNotes(), []);
```

Dependency array `[]` artinya hanya di-compute sekali saat mount. Kalau user buat catatan baru terus kembali ke dashboard tanpa full refresh, ReflectionCard mungkin nampilin data lama.

**Fix:** Tambahkan listener event `madad:refresh`:
```js
const [, setTick] = useStateC(0);
useEffectC(() => {
  const h = () => setTick(t => t + 1);
  window.addEventListener("madad:refresh", h);
  return () => window.removeEventListener("madad:refresh", h);
}, []);
const note      = useMemoC(() => getReflectionOfTheDay(), [tick]);
```
Tapi sebenarnya ini edge case — refleksi seed-by-date, gak akan ganti dalam hari yang sama anyway. Low priority.
